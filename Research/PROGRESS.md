  **Subagent #17 (image/flashing mechanics) returned — decisive, well-evidenced findings, no repo changes yet:**
  - **The load-bearing question, answered with high confidence via primary sources, not general knowledge**: does the prebuilt Arduino-ESP32 second-stage bootloader require an ESP-IDF-style app-descriptor header a from-scratch app can't easily produce? **No.** Confirmed four ways: the documented image-verify contract (magic byte/checksum/hash/signature only) has no descriptor requirement; the real ESP-IDF v4.4.7 source (fetched and read, matching this framework's exact version) gates descriptor-checking behind `CONFIG_BOOTLOADER_APP_ANTI_ROLLBACK`, off by default; running `xtensa-esp32-elf-nm` directly on the actual local `bootloader_qio_40m.elf` shows the descriptor-checking function isn't even linked into this specific prebuilt binary; and the actual jump sequence in `bootloader_utility.c` is a raw function-pointer call to the image header's entry address, no descriptor lookup. This was the single biggest open risk and it's resolved clean.
  - **Real landmine caught before it bit**: the prebuilt `tools/partitions/default.bin` does NOT match `tools/partitions/default.csv` in this exact framework version — verified by regenerating from the CSV and byte-diffing (differ at byte 132; the prebuilt has a stale `eeprom` partition where the current CSV has `spiffs`+`coredump`). **Must regenerate `partitions.bin` from the CSV via `gen_esp32part.py`, never use the prebuilt `default.bin` directly.**
  - `elf2image` mechanics confirmed empirically against the real `examples/esp32_minimal` ELF (built via `cmake --preset esp32-xtensa`): entry point is a raw copy of `e_entry` (no symbol lookup, `ENTRY(_start)` is sufficient); `.bss` sections are read via the ELF section table and **completely excluded from the output image** (routed to a `nobits_sections` list, never written) — direct confirmation that whatever currently lives in `.bss`-mapped RAM at boot is genuinely uninitialized garbage unless startup code zeroes it, not just a style nitpick; no address-range validation happens at `elf2image` time (succeeded silently converting the current, unmodified, IRAM-only-layout ELF) — real-address correctness is instead checked later, at flash-time, by the bootloader itself (`verify_load_addresses()`, which only rejects overlap with the *bootloader's own* reserved regions — a non-issue for an example this small). Do NOT pass `--elf-sha256-offset` (that's for ESP-IDF apps with a `.flash.appdesc` section reserved for it; passing it against a from-scratch ELF hits a real, confirmed "not all zero, refusing to overwrite" fatal error).
  - Bootloader variant: `dio`+`40m` is the correct safe default for an unbranded/generic board (confirmed via this exact machine's own `esp32dev.json` board definition AND PlatformIO's own qio→dio/qout→dio downgrade-for-unknown-boards logic).
  - Flash offsets verified against the real, current `default.csv` (not assumed): bootloader `0x1000`, partition table `0x8000`, `boot_app0.bin` `0xe000` (needed — the CSV does include an `otadata` partition), app image `0x10000`. Full working `write_flash` command recorded in commit `f4a09c6`.
  **Subagent #18 (startup-code/memory-layout correctness) returned — found a real, hardware-breaking bug, then built and verified a fix (see commit `e438462` for the full implementation).** Empirically confirmed via `nm` on the real linked ELF that `scenario.hpp`'s global object graph needs zero dynamic C++ initialization (no `_GLOBAL__sub_I_*`, no `.init_array`/`.ctors` content) — just `.bss` zeroing and a `.data` load-to-run copy. But proved (via a real disassembled instruction, not theory) that the old flat `0x40080000` single-region layout placed `.data`/`.bss` — including several `bool` globals — inside IRAM0, the word-only-accessible instruction bus: a byte-store inside `on_battery_level()` targeting an IRAM0 address, which would raise a `LoadStoreError` on real silicon. Built, compiled, and linked a fix (separate IRAM0/DRAM0 memory regions, real LMA≠VMA `.data` split, `.bss` zeroing, defensive `.ctors` walk) and confirmed via `readelf -l` + re-disassembly that it actually closes the bug. **This fix was independently re-verified directly against the real repo files (not just trusted from the subagent's scratch copy) and landed in commit `e438462`**: `cmake --build --preset esp32-xtensa` compiles/links clean, `readelf -l` shows the correct LMA≠VMA program headers, and the previously-faulting store now resolves to `0x3ffb00ad`, `g_battery_low`'s real DRAM0 address.
  **First real flash attempt: succeeded completely.** Converted the fixed ELF + the real, already-installed Arduino-ESP32 prebuilt `bootloader_dio_40m.elf` + a freshly-regenerated `partitions.bin` (not the stale prebuilt) + `boot_app0.bin` into a full image set and flashed all four components to the real board at the verified offsets — every segment written and hash-verified (`Chip is ESP32-D0WD-V3 (revision v3.1)`, `MAC: 8c:94:df:93:b4:50`). Separately confirmed via `flash_id`: 4MB flash detected (matches what was assumed), "flash voltage set by a strapping pin."
  **But the board doesn't actually boot into the app — a real, reproduced boot loop.** Captured real boot output over serial (pyserial, 115200 baud, confirmed reproducible byte-for-byte across two independent flash+observe cycles with zero manual DTR/RTS interference, so not a test-methodology artifact): the ROM bootloader successfully loads the prebuilt second-stage bootloader and jumps to its entry point (`0x400805e4`) — consistently, correctly — then total silence for many seconds, then `rst:0x10 (RTCWDT_RTC_RESET)` (an RTC watchdog firing) and the whole sequence repeats. The bootloader itself never produces a single byte of its own log output before the watchdog resets it, which is not normal — a real ESP-IDF/Arduino bootloader logs almost immediately. **This is very likely a problem in how the reused, prebuilt Espressif bootloader binary was converted/packaged, not in Lw-Eh's own code or the just-verified linker/startup fix** (which only affects `firmware.bin`, loaded later in the boot chain — the hang happens before that point is ever reached).
