Two quick preliminaries first: confirmed `bloaty` genuinely isn't installable-for-free the way the Xtensa toolchain turned out to be (checked `winget list`, scoop, chocolatey, and the PlatformIO package cache directly — all empty; a real install would need explicit permission, not pursued) — this closes out an option flagged as open for two firings running. Then opened `build/esp32-xtensa/size_audit/lw_eh_size_audit_with_lweh_host` (the real Xtensa binary from firing 8's 520-byte measurement) in the IDA Pro MCP tools available in this environment, cross-checked against direct `readelf`/`objdump`/`nm` inspection with the real toolchain. Findings, one per loop-checklist item: **(1) struct layout** — computed `sizeof` for every Lw-Eh type from real global-variable address deltas (not just the host `static_assert`): `intrusive_signal<connection_event>` = 4 bytes (one pointer, exactly as designed), `signal<Event,4>` = 32 bytes (4 × 8-byte `delegate`, confirming `sizeof(delegate) == 2 pointers` on real 32-bit hardware), `sensor_logger` = 8 bytes with zero padding, `connection_logger` = 12 bytes (vptr + inherited `next` + `bool`, padded to 4-byte alignment) — every one matches source-level reasoning exactly, no surprises. **(2) thin-shim inlining** — confirmed `attach()`/`bind()`/`detach()`/`publish()` produce *zero* standalone symbols in the binary; `main` is classified as a leaf function (0 explicit callees) with all of `run_scenario()`'s three attach+publish sequences fully inlined into it, leaving only the three necessarily-address-taken per-binding stub functions (`free_function_stub`, `member_function_stub`, `invoke_erased`) as separate, small (11-52 byte) symbols. **(3) RTTI/exception artifacts** — confirmed zero `_ZTI...` (typeinfo) symbols anywhere in the binary; confirmed `.eh_frame` (4 bytes) is the standard empty zero-length CIE terminator (`00000000`), not real unwind metadata. Dumped the actual vtable bytes for `connection_logger` and found something worth recording precisely: the Itanium ABI's 2-word vtable header (offset-to-top + typeinfo pointer) survives *even under* `-fno-rtti` — it's just zeroed rather than pointing to a real typeinfo object. `-fno-rtti` eliminates the typeinfo *object* and its support code, not this fixed ABI-mandated header slot — a sharper, now-precisely-confirmed version of the "~16 bytes for a vtable" estimate from firing 8. **(4) COMDAT/folding** — confirmed no redundant duplicate instantiations survived; each of the three stub functions is genuinely distinct (different `Event`/signature) and appears exactly once. **(5) cache-line/alignment** — explicitly judged not applicable at this scale (the whole allocated image is ~875 bytes; cache-line-level analysis is meaningful for larger, hot-loop-heavy code, not a sub-1KB firmware image, so this wasn't forced). **(6) `--gc-sections` effectiveness** — re-linked with `-Wl,--print-gc-sections` and found it removed exactly one thing: an empty `.data` section from the toolchain's own `crtbegin.o` startup object, unrelated to Lw-Eh. Nothing from Lw-Eh's own object file needed discarding — the compile/LTO stage is already doing the real dead-code elimination; the linker pass is confirmation/insurance, not heavy lifting. Also independently verified, by summing only the `ALLOC`-flagged sections from `objdump -h` (`.text`+`.rodata`+`.eh_frame`+`.ctors`+`.dtors`+`.data`+`.bss` = 703+16+4+8+8+24+112), that the total comes to exactly 875 bytes — matching `size_report.py`'s "with-lweh" number exactly, and confirming the DWARF debug sections and Xtensa metadata sections it also emits are correctly non-`ALLOC` and don't inflate the measured/flashed size.
