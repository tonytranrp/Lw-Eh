- **Current measured size:** Host-proxy probes (control vs. 1-event-type vs. 2-event-type, all built on this machine's Clang/llvm-mingw, x86-64 Windows PE): **with `-flto=full`** — 1st event type costs +264 bytes over baseline (unchanged from last firing), 2nd event type costs only **+136 bytes marginal** (400 total − 264). **Without `-flto`** (closer to what the actual embedded GCC toolchains will do by default, per last-last firing's finding that none of them enable LTO out of the box) — 1st event type costs +328 bytes, 2nd event type costs **+176 bytes marginal** (504 total − 328). In both cases the second event type's marginal cost is roughly **half** the first's (52% with LTO, 54% without), not a full duplicate — meaningful evidence that the current fully-templated-per-event-type `signal<Event,N>` design (no explicit `detail::signal_core` sharing yet) scales sub-linearly on its own, likely via a mix of genuine COMDAT/instruction-level overlap the optimizer finds despite the two `Event` types differing, and some one-time fixed cost (e.g. runtime support glue triggered once) amortized across the first type but not repeated by the second — this split hasn't been attributed via disassembly, so treat "why" as partially understood, not fully explained. Reassuring for the deferred-core-extraction decision specifically because the sub-linear scaling holds even *without* LTO, so it's not purely an artifact of this host toolchain's aggressive optimization. Still caveated the same way as last firing: this is x86-64 Windows PE data, not the real 32-bit embedded ISAs, and only 2 data points (N=1, N=2) — a 3rd or 4th event type would meaningfully strengthen or weaken the "sub-linear, not flat" read (flat marginal cost per type would suggest genuine amortized-sharing; a returning-to-linear 3rd-type cost would suggest the 2nd type's discount was mostly the one-time-overhead explanation, not real sharing). `size_audit/`'s real CMake wiring is still an unfixed, non-urgent follow-up (unchanged from last firing).
