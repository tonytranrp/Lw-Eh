- **Last completed:** Fixed `size_audit/`'s CMake wiring — deferred three firings running, this closes it out. Root cause of the original brokenness (identified precisely, not just patched around): (1) `size_audit/`'s "with_lweh" target only ever existed by reusing `examples/esp32_minimal`'s `app_main()`-based, `-nostdlib`/custom-linker-script-dependent build, which simply cannot link on a native host target (no `main()` symbol, wrong entry-point convention) — so the `size-host` preset from Phase 0 never actually worked despite existing since firing 1; (2) `size_audit/control_main.cpp` had the exact same problem (`app_main()`, no `main()`) and, on inspection, was *never actually correctly wired even for the embedded case either* — `lw_eh_size_audit_control`'s CMake target never applied `-nostdlib`/a linker script the way `examples/*` do, so it always expected a normal `main()` entry point regardless of target, a pre-existing inconsistency only surfaced while fixing the host-blocking issue. Fix: extracted `examples/esp32_minimal/main.cpp`'s usage logic into a new shared `examples/esp32_minimal/scenario.hpp` (two event types, free-function + member-function listeners, `run_scenario(pin_input, sensor_input)`) so the embedded example and a new host-buildable probe (`size_audit/with_lweh_main.cpp`, real `int main()`, volatile-sourced inputs to defeat constant-folding) can never silently drift apart — the "one source of truth for realistic usage" principle from `Research/ARCHITECTURE.md`, now actually holding structurally instead of just in documentation. Fixed `control_main.cpp` to a plain `int main() { return 0; }`. Updated `size_audit/CMakeLists.txt` to build both `lw_eh_size_audit_with_lweh_host` (always, no toolchain needed) and the real embedded target (still gated on `LWEH_BUILD_EXAMPLES` + a real toolchain, unchanged). Fully implemented `size_report.py` (previously a `TODO` stub since Phase 0) — resolves `size`/`nm` tools with a fallback from the cross-prefixed name to `llvm-`-prefixed names (needed since this dev machine has LLVM tools, not plain GNU binutils, on PATH), finds built executables by searching the build directory rather than assuming a fixed layout, computes the incremental delta from the `size` tool's berkeley-format "dec" column (the same number this project's manual probes have been reading by hand since firing 2), prints a top-15-by-size `nm` symbol table, and runs `bloaty` diff mode if it's on PATH (confirmed not installed on this machine — gracefully skipped, not a hard failure). Ran the whole pipeline for real: configured `size-host`, built both targets, ran the script — it works end-to-end, no more ad hoc scratchpad reconstruction needed going forward.

Verified via `nm --size-sort --print-size -C` that Lw-Eh's own compiled symbols (`free_function_stub`/`member_function_stub` instantiations, the global signal/logger objects) are present, correctly named, and small enough that none of them make the top-15-by-size list — dominated entirely by unavoidable C-runtime functions (`malloc`, `memcpy`, `vfprintf`, `mainCRTStartup`), a clean illustration of research.md §B6's "platform runtime floor" concept. One caveat noted for whoever reads the `nm` output on this host: PE/COFF (this Windows target) doesn't populate per-symbol size fields the way ELF does, so `--print-size` shows `00000000` for everything here — that's a host-platform-format limitation, not a script bug; a real embedded (ELF) build will show real per-symbol sizes through the same script unchanged. 424 bytes is comfortably within the 1-2KB budget. Standing caveat, unchanged: this is still x86-64 Windows PE host-proxy data, not the real 32-bit embedded ISAs — that gap is now the single most significant one left (see "Next up").
