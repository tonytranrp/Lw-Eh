This is the durable, git-tracked ledger for the autonomous build loop. Updated at the end of every firing. Git history is ground truth if this ever drifts — see `git log --oneline`.

- **Phase:** embedded-validation (first real embedded cross-compile achieved this firing; correctness testing remains host-only by design since binaries can't execute here)
- **Last completed:** Investigated whether a genuinely usable embedded toolchain exists on this machine, per last firing's plan — and found one. The original "no toolchain" conclusion (firing 1) was a `where.exe` check for a few expected binary names, which don't match PlatformIO's non-PATH install layout, and was never re-verified since. Found a real, working `xtensa-esp32-elf-gcc`/`g++` 8.4.0 (crosstool-NG `esp-2021r2-patch5`) at `~/.platformio/packages/toolchain-xtensa-esp32/bin/`, left over from a prior, unrelated PlatformIO setup — an already-present resource, not something requiring a fresh download/install (which would have needed the user's explicit permission and wasn't attempted). Persisted this correction to durable memory so future sessions don't re-assume the toolchain gap. Prepended it to `PATH` for this session's build commands only (kept `cmake/toolchains/xtensa-esp32.cmake` itself unmodified/portable, matching its documented "assume it's on PATH" contract) and attempted the first-ever real cross-compile of an Lw-Eh example. Found and fixed three real, previously-undetectable bugs along the way (undetectable because `LWEH_BUILD_EXAMPLES` had never actually been turned on with a working toolchain before): (1) `examples/CMakeLists.txt` didn't exist at all — the root `CMakeLists.txt`'s `add_subdirectory(examples)` had nothing to add; created it, gated per-architecture on `CMAKE_SYSTEM_PROCESSOR` so a single active toolchain doesn't try to compile-and-link the other three architectures' examples with the wrong compiler/linker script. (2) The root `project()` only declared `CXX`, so `.S`/`.c` startup files were silently never compiled — zero build steps, no error, and `--gc-sections` then garbage-collected the *entire* example (including all of Lw-Eh's own code) down to a genuinely empty 0-byte binary, which would have been silently reported as a valid measurement if not caught. Fixed by scoping `enable_language(ASM)`/`enable_language(C)` into `examples/CMakeLists.txt` itself (not the root), so host-only configurations never pay for compiler detection they don't need. (3) The real link failure once ASM was fixed — `dangerous relocation: l32r: literal placed after use` — was mis-diagnosed at first as an LTO/Xtensa interaction (a documented, real risk from firing 2's own prior research), leading to an LTO-disabling fix that turned out to be treating a symptom: the actual root cause was `esp32.ld` never handling `.literal*` input sections at all, so the linker's orphan-section placement put them after the code that referenced them, violating Xtensa's l32r instruction's backward-only addressing constraint. Confirmed by direct empirical iteration (not guessing, not a subagent dispatch — this was a fast, directly testable question with the real toolchain in hand): fixing the linker script (`*(.literal*) *(.text*)` in that order, in the `.text` output section) resolved the link failure with LTO *disabled*; re-testing with LTO *re-enabled* against the fixed script also worked (and was ~3% smaller), confirming LTO was never actually the problem and reverting the unnecessary disable. `examples/esp32_minimal` now compiles, links, and produces a valid, non-empty ESP32 ELF for the first time in this project's history.
- **Current measured size:** **372 bytes incremental** — the first real, checked-in-tooling-derived embedded measurement this project has (previously all numbers were host-proxy). Obtained by running the actual `size_audit/size_report.py` (unmodified from last firing) against genuine `xtensa-esp32-elf-gcc` output: `python size_audit/size_report.py --build-dir build/esp32-xtensa --prefix xtensa-esp32-elf-`. Discovered along the way that `lw_eh_size_audit_control`/`lw_eh_size_audit_with_lweh_host` (the host-proxy pair from last firing) happen to cross-compile correctly for Xtensa too, since they don't hardcode anything host-specific — so the exact same script, unmodified, produced a genuine embedded number without needing the separate `lw_eh_example_esp32_minimal`-reuse path at all. Cross-checked against the `examples/esp32_minimal` build directly (via the `-nostdlib`/custom-linker-script path): 343 bytes total (267 text + 76 bss) — a different absolute baseline (near-zero, since almost nothing is linked by default) but a consistent order of magnitude. `nm` on the real ELF shows actual per-symbol sizes for the first time (PE/COFF on the host never populated these) — `g_button_signal`/`g_sensor_signal` are exactly 32 bytes each (4-slot array × 8-byte `delegate<Sig>`, i.e. 2×4-byte pointers per slot on this 32-bit target), directly confirming the `sizeof(delegate) == 2 pointers` invariant holds on real embedded hardware, not just via the host `static_assert`. Both numbers are comfortably within the 1-2KB budget, with the real one (372 bytes) *smaller* than the best host-proxy estimate (424 bytes from last firing) — consistent with 32-bit vs. 64-bit pointer width and a denser target ISA. RISC-V, ARM, and AVR toolchains were searched for but not found anywhere on this machine — those three targets remain unvalidated.
