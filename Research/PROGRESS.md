- **Last completed:** Fresh orientation surfaced `LWEH_HAS_CONCEPTS` (`config.hpp`) as genuinely unused anywhere in the codebase — pure Phase-0 scaffolding, never acted on. Rather than either leaving it or guessing whether to use it, treated this as a real engineering decision (touches the public API contract of `delegate<Sig>`, the most foundational, most-tested type in the library) and dispatched 2 parallel subagents per the loop's investigate-before-implementing rule: one on the pure technical correctness of a proposed C++20 `requires`-clause fix for `bind<Fn>()`'s diagnostic quality, one on whether it's worth doing at all against this project's demonstrated minimalism bias (explicitly benchmarked against firing 19's `signal_core` deferral — a *stronger* candidate that still lost).
  Both converged on the same practical conclusion via independent reasoning, which is strong, well-triangulated evidence: **skip `requires`/concepts entirely, use a plain C++17 `static_assert(std::is_invocable_r_v<...>)` instead.** The evaluation subagent's case: `requires` only helps C++20 consumers (research.md §A8 treats C++17 as the *guaranteed* floor), adds permanent dual-path complexity to the tightest-invariant file in the library, and doesn't clear the bar firing 19 set for a *stronger*, measured, fully-specified candidate. The correctness subagent's case, independently and more surprisingly: `is_invocable_r_v` — used *either* in a `requires`-clause or a plain `static_assert`, it doesn't matter which — has its own real loophole, because it's built on the identical `INVOKE` semantics `std::invoke` has, which `delegate.hpp`'s own existing comment already documents avoiding on purpose (direct-call syntax in `free_function_stub`, specifically to dodge `std::invoke` treating a mistakenly-passed member pointer as an "unbound member" call). A member-function pointer whose shape happens to match "callable with `Args...[0]` as the implicit object" slips past `is_invocable_r_v` undetected — meaning a naive `static_assert(is_invocable_r_v<...>)` alone would NOT have been a strictly-better replacement for the existing, precise `is_member_pointer_v` check; it would have been a *different* check with its own gap. Also found, independently confirmed via actual clang compilation (not just reasoned): when a `requires`-clause and a body `static_assert` both could apply to the same mistake, SFINAE removes the failing candidate before the body is ever reached — meaning the `requires`-clause's generic "constraints not satisfied" message would have **pre-empted and hidden** the existing, friendlier `static_assert` text for the common mismatch case, a real diagnostic-quality regression, not just unnecessary complexity.
  **Implemented accordingly, precisely, not a blind combination of both subagents' raw recommendations**: added a NEW `static_assert(std::is_invocable_r_v<Ret, decltype(Fn), Args...>, ...)` to `bind<Fn>()` itself (catches wrong arity/parameter-type mismatches early, at the call site, before `free_function_stub<Fn>` is even instantiated) while leaving the EXISTING `is_member_pointer_v` check in `free_function_stub`'s body completely untouched (it's the only complete backstop against the invocability loophole, being a pure type check immune to it — traced through the actual case-by-case logic to confirm the two checks are genuinely complementary, not redundant, before writing anything). Added the equivalent new check to `bind<MemFn>(T*)` too — this one closes a real, previously totally-unguarded gap (the member overload had zero compile-time signature checking at all before this), with its own narrow, honestly-documented limitation (a free function shaped exactly like `Ret(T*, Args...)` could still slip past it) recorded in the code itself rather than silently ignored, matching how `delegate`'s own equality operator already documents its ICF-folding caveat. Removed `LWEH_HAS_CONCEPTS` from `config.hpp` — unanimous across both subagents, no consumer anywhere, sat unused since the project's very first commit.
