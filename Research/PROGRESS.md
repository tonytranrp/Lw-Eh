- **Phase:** embedded-validation (unchanged — codegen deep-verification pass, no source changes this firing)
- **Last completed:** Picked up last firing's candidate (a): confirm precisely which mechanism folded `signal<battery_level_event,4>::publish()` and `signal<wifi_rssi_event,4>::publish()` into one shared copy, and check whether `attach()`/`detach()` get the same treatment. IDA's idalib MCP tools were connected and correctly recognized the Xtensa ELF (`survey_binary` cleanly identified all 4 segments, 9 functions, entry points) — used directly rather than falling back to objdump/readelf. Disassembled `app_main` in full (134 instructions, matching `run_scenario()` fully inlined into it) and the shared `publish()` function itself. Findings, precise rather than speculative:
  1. **`attach()` is fully inlined into `app_main` for all five calls (button, sensor, connection, battery, wifi) — it never exists as a standalone function anywhere in the binary.** `total_functions` is 9 and none of them is `attach()`; the disassembly shows five distinct "scan 4 slots for an empty one, write obj_/stub_" (or, for `connection_event`, the `intrusive_core::link()` head-insertion sequence) code blocks inlined directly into `app_main`, none sharing code with any other. Since IPA-ICF (and any other whole-program folding pass) operates on distinct function bodies/symbols, code that's already been inlined away is invisible to it — `attach()`'s cost is paid in full, every single time, with **zero** folding, structurally identical across calls or not.
  2. **`publish()`/`dispatch()` is a mixed picture, not uniformly inlined or uniformly shared.** `button_event`'s and `sensor_reading_event`'s dispatch loops (and `connection_event`'s, via `intrusive_core::dispatch()`) are also fully inlined into `app_main` as real loop structures (backward branches, not unrolled) — the first two `signal<>` instantiations and the one `intrusive_signal<>` instantiation encountered in source order all got inlined outright, same as every `attach()` call. But `battery_level_event`'s and `wifi_rssi_event`'s `publish()` calls both emit a real `call8` instruction **to the exact same address**, `_ZN4lweh6signalIN12lweh_example19battery_level_eventELj4EE7publishERKS2_` — confirmed directly in `app_main`'s own disassembly, not inferred from a post-link address coincidence. Disassembling that shared function confirms *why* it's safely shareable: its machine code only computes `this+ (MaxListeners*sizeof(delegate))` as a loop bound (baked in as the literal `0x20` = 4×8 here) and passes the event pointer through completely opaquely (`a3` is never dereferenced or interpreted) — `Event`'s own type/layout never enters this function's codegen at all, so *any* `signal<Event,4>` instantiation compiles to byte-identical code, regardless of what `Event` actually is.
  3. **The mechanism is very likely GCC's IPA-ICF pass, enabled by `-flto`** (`lw_eh_size_flags`), not a linker-level COMDAT/section merge — the shared call target is embedded directly in `app_main`'s own compiled instructions on both call sites, meaning the *compiler's* whole-program optimizer treated the two instantiations as interchangeable, not a linker folding two already-emitted, already-addressed sections after the fact. This lines up with subagent investigation #5's standing caveat (Phase 1) that ICF-style folding "isn't automatic on Lw-Eh's target toolchains without `-flto`/gold/lld" — this project does use `-flto`, and this firing is the first direct, disassembly-level confirmation that the caveat's positive case (folding *does* happen when `-flto` is on) is real, not just theoretically possible.
  4. **Practical implication for the deferred `signal_core` question**: the ~143-146 bytes/type marginal cost measured in firings 16-17 is dominated by `attach()`'s fully-duplicated inlined code and each type's `free_function_stub` trampoline (which correctly can *never* fold across types, since each one's `template<auto Fn>` NTTP bakes in a genuinely different call target) — **not** by `publish()`, which the compiler is already mostly recovering sharing for on its own once a third-plus instantiation exists. If the deferred extraction in `detail/signal_core.hpp` is ever revisited, `attach()` (and presumably `detach()`, structurally identical in kind but not directly exercised by the current scenario — `run_scenario()` never calls `detach()`) is the higher-leverage target, not `publish()`, which is already close to "for free" past the first couple of event types. Not a recommendation to do the extraction now — 730 bytes total for 5 event types is still comfortably inside the 1-2KB budget — just a sharper, evidence-based answer for *when* that question comes up again, rather than a source-level guess.
