- **Phase:** hardening (unchanged)
- **Last completed:** Fresh orientation noticed a real, if narrow, documentation asymmetry: `detail::intrusive_link`'s destructor (`detail/intrusive_core.hpp`) carries an explanatory comment for why it's protected and non-virtual ("nothing ever deletes through an `intrusive_link*`, only through the most-derived listener type the caller owns"), but `intrusive_node<Event>`'s destructor (`intrusive_signal.hpp`) — the actual type real consumers inherit from and hold — uses the identical protected+non-virtual pattern with **no comment at all**. Every other non-obvious safety property in this pair of files (the no-auto-detach tradeoff, the reentrancy caveat) is carefully documented; this one, arguably more fundamental (it's the property that makes it safe to inherit from `intrusive_node<Event>` without paying for a vtable-based virtual destructor), wasn't.
  Also noticed the *existing* comment on `intrusive_link`, while directionally correct, understates the actual guarantee: "nothing ever deletes through X" reads like a design convention callers are trusted to follow, when the real guarantee is stronger — `protected` access makes deleting through the base pointer from outside the class hierarchy a **compile error**, enforced by the language itself, not a discipline anyone has to remember. Verified this empirically rather than trusting textbook C++ knowledge alone (this project's own established standard, applied even to something this basic): wrote a throwaway snippet deliberately deleting an `intrusive_node<Event>*` obtained via `new` on a concrete listener type, compiled it directly — got exactly `error: calling a protected destructor of class 'lweh::intrusive_node<my_event>'`, plus an unprompted bonus: Clang's `-Wdelete-abstract-non-virtual-dtor` warning, meaning even a hypothetical public-but-non-virtual version of this destructor would still get flagged by the compiler as suspicious.
  Fixed both: added the missing, detailed comment to `intrusive_node<Event>`'s destructor (citing the empirical confirmation directly), and sharpened `intrusive_link`'s existing comment to state the compile-time-enforced guarantee precisely rather than the weaker "nothing ever does X" framing, cross-referencing each other. Comment-only change — no logic touched in either file.
