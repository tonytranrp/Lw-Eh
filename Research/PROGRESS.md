  **Real disassembly findings (`xtensa-esp32-elf-nm`/`objdump` directly on the Xtensa .elf):** `lweh::detail::intrusive_core::dispatch()` exists as exactly ONE shared, non-template symbol — real confirmation the dispatch/`publish()` half of the "genuinely shared engine" claim holds, the same way `signal<>::publish()` was shown to fold in firing 18. However `link()` (the body of `attach()`) does **not** manifest as any standalone shared symbol at all — disassembly of `app_main` shows it fully inlined at each of the two `intrusive_signal<Event>::attach()` call sites (the exact 3-instruction `node.next = head_; head_ = &node;` pattern appears twice, independently), the same "always inlined, never shared" behavior firing 18 found for `signal<>::attach()`. So the doc comment's sharing claim is accurate for dispatch, but doesn't extend to attach/detach the way its wording implies — noted as a real, previously-unverified nuance, not yet fixed in the doc comment (see Next up).
  Separately, `intrusive_node<motion_event>::invoke_erased()` and `intrusive_node<connection_event>::invoke_erased()` disassembled to **byte-for-byte identical** machine code (manually diffed instruction-by-instruction, confirmed identical) yet were placed at two different addresses, NOT folded — while in the same binary, `motion_logger::on_event()` and `connection_logger::on_event()` (also byte-identical, also referenced only from two different vtables) WERE folded to one shared address. This asymmetry (identical situation, different ICF outcome) isn't explained by anything in research.md, so a subagent investigation was launched into GCC 8.4.0's `-fipa-icf` behavior specifically.
  **Subagent's answer (high confidence on the mechanism, medium on the exact GCC-8.4.0-source micro-detail):** built a faithful, self-contained repro on this project's actual toolchain with the actual flags plus `-fdump-ipa-icf-details`, and confirmed its own repro reproduces the identical pattern (same symbols fold/don't-fold, same disassembly) before trusting the dump. IPA-ICF is two-stage: a cheap structural hash buckets candidates first, and the expensive pairwise `equals()` body-walk only ever runs within a bucket. The dump's hash-grouping pass put both `on_event` instantiations in one bucket from the start (later confirmed via an explicit `"Equals called for:"` log line); `invoke_erased`'s two instantiations landed in different buckets immediately and the pairwise comparator was never even invoked for that pair — grep for `"Equals called for:"` in the dump returns exactly 2 hits (`on_event`, and `publish`/`dispatch`) and 0 for `invoke_erased`. A clean internal control group ruled out the two obvious alternate theories: `intrusive_signal<Event>::publish()` also contains a call (to shared `intrusive_core::dispatch()`) and folds fine, so "containing any call" doesn't disqualify folding; and both `on_event`/`invoke_erased` are equally vtable-address-taken, so that's not it either. What's specific to `invoke_erased`: its call to `on_event` is *polymorphic* (indirect/virtual), and GCC's structural hash (per current GCC trunk source, not the exact 8.4.0 tag — hence "medium" not "high" on this specific point) hashes a plain direct call "thin" (by tree-code only) but a polymorphic callee more richly (type-bearing), letting the per-`Event`-type distinctness leak into the hash before any body comparison gets a chance to notice the bodies are identical.
  **Practical implication, per the subagent, and independently endorsed here:** not realistically fixable while keeping the current two-layer design (type-erased `invoke_erased` thunk → the user's virtual `on_event`) — the polymorphic call is exactly the mechanism the library's own doc comment credits for erasing `Event` "for free." A fix would mean collapsing the thunk and the user's override into one virtual function per listener, a real API/ergonomics tradeoff, not a local tweak, to save on the order of tens of bytes total (scales with distinct Event *types*, not listener instances). **No code change made** — matches the pre-registered scoping ("no code action expected regardless of the answer"); this stays a documented, non-actionable GCC-ICF conservatism.
