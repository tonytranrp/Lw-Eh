- **Last completed:** Did a genuinely fresh, full re-read of `Research/research.md` end-to-end (last one was firing 11, 10 firings ago) rather than defaulting to a standing candidate, per firing 20's own advice. Found something concrete and actionable that hadn't been acted on: this project's whole reentrancy-safety design (`signal<>`'s mark-and-skip contract, `intrusive_signal<>`'s singly-linked-list contract) was originally validated via one-off, ASan/UBSan-compiled adversarial stress-test programs during Phase 0 investigation (firings 8-9) — but the **standing** CTest suite that runs every single firing has never itself run under any sanitizer. That's a real, permanent gap: a future code change introducing a subtle memory-safety or UB regression wouldn't necessarily crash outright, and all 7 tests could keep passing while silently harboring it.
  Added a `LWEH_SANITIZE` CMake option (default `OFF`) and a new `host-asan` preset that builds the exact same 7-test suite with `-fsanitize=address,undefined -fno-omit-frame-pointer`. Deliberately scoped to `tests/`'s individual targets, **not** added to `lw_eh_size_flags` itself — that INTERFACE target is shared by `examples/*`'s cross-compiled embedded binaries, which have no sanitizer runtime available for bare-metal targets; blindly adding sanitizer flags there would have broken every embedded cross-compile preset. Ran the new preset for real: **all 7 tests pass clean under ASan+UBSan** — a genuine, first-time confirmation that the library is actually memory-safe and UB-free across its whole test surface, not just "happens to not crash on this particular compiler." Confirmed zero impact on the pre-existing `host-debug` preset (`LWEH_SANITIZE` defaults `OFF`, byte-identical compile options, "ninja: no work to do" on rebuild, all 7 still pass in the same ~0.25s) and zero impact on the real Xtensa embedded build (375 bytes incremental, unchanged, "no work to do" on rebuild) — confirming the addition is cleanly isolated to its own opt-in path. Documented the new preset in `README.md`'s build/test section.
  While re-reading research.md fresh, also specifically checked a few other B4/B9 items against the current codebase to see if anything else was similarly unactioned: `-mtext-section-literals` (Xtensa) — correctly NOT set anywhere, matching research.md's own recommendation to keep the default separate-literal-section behavior for cross-TU dedup; `-fshort-enums` — moot, the library defines no enum types in its public API at all; `-Wl,--relax` for architectures beyond Xtensa — already correctly gated on `LWEH_IS_GNU` (not Xtensa-specific), so it already applies to any GNU-family cross-compiler (RISC-V/ARM/AVR GCC included) per `cmake/LwEhAggressiveFlags.cmake`'s existing logic, not something needing a fix. Two items flagged as real but smaller and not acted on this firing (see Next up): `-Wl,--print-gc-sections` has never actually been run against the real Xtensa build to *confirm* `--gc-sections` is eliminating what's expected (only assumed via the flag being set); `-mlongcalls` (Xtensa, ESP-IDF's own default) isn't currently set in `cmake/toolchains/xtensa-esp32.cmake` — harmless at the library's current sub-1KB size (calls can't be out of range yet) but worth adding for completeness/future-proofing.
