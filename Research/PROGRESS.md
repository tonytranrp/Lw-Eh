# Lw-Eh Progress Ledger

This is the durable, git-tracked ledger for the autonomous build loop. Updated at the end of every firing. Git history is ground truth if this ever drifts — see `git log --oneline`.

- **Phase:** core-implementation (delegate + signal done; multi-event-type coverage, intrusive storage, and real embedded validation remain)
- **Last completed:** Implemented `lweh::signal<Event, MaxListeners>` (`include/lweh/signal.hpp`) — fixed-capacity, zero-allocation, multi-listener dispatch for one event type. Investigated the one open design question first (per last firing's plan) with 2 parallel subagents: one read EnTT's actual `sigh`/`sink` source directly and traced its real reentrancy contract, one adversarially stress-tested (compiled, ASan/UBSan-checked) a proposed "mark-and-skip" alternative. Result was decisive, not a coin-flip: EnTT's real removal mechanism is swap-with-last+`pop_back()` (not shift-down as research.md's citation implied), and combined with reverse iteration it has a **confirmed, reproducible double-invoke bug** — and in compounding cases an out-of-bounds `operator[]` access, i.e. real UB — when a listener detaches a not-yet-visited peer. Mark-and-skip (occupancy tracked per-slot via `delegate`'s own `operator bool()`, forward iteration, `attach()` first-fits an empty slot, `detach()` just clears the slot, no compaction ever) had zero failures across every scenario tested (self-detach, detach-other before/after cursor, attach-during-dispatch) and is strictly safer. Implemented signal on top of this, added `operator==`/`operator!=` to `delegate<Sig>` (needed so `detach<Fn>()`/`detach<MemFn>(T*)` can find the matching slot by reconstructed value, mirroring `bind`'s own syntax per the Phase 0 architecture decision) with a documented narrow caveat about ICF-folding making two different-but-identical-code member functions on the same instance compare equal (not a real risk for detach-by-value, flagged for the record). Made and documented one real architecture deviation: `detail::signal_core`'s event-erased dispatch-sharing role from research.md §A4 is deferred (not abandoned) — genuinely erasing `Event` would need an extra adapter-stub layer for a size win §A4 itself says only pays off "past small single digits" of event types, which this codebase doesn't have yet; recorded as a Phase-0 amendment in `ARCHITECTURE.md` with reasoning. Rewrote all 5 previously-placeholder test files (`attach_detach`, `dispatch_order`, `capacity_boundary`, `reentrancy`, `multi_event_isolation`) with real assertions — found and fixed 2 genuine bugs along the way, both in the new test code, not the library: a reentrancy test that didn't account for a persistent listener re-triggering its one-shot-looking detach+attach logic on every subsequent `publish()`, and a plain arithmetic slip in an expected value. Also caught and fixed, before it could bite on a real embedded GCC, a portability trap of my own making: I'd bound several test-only member functions declared on **local classes** (defined inside `main()`) as `template<auto Fn>` NTTP arguments — local classes lack linkage, which is exactly the class of NTTP pitfall last firing's investigation was about; happened to compile fine on this host's Clang but isn't something to rely on, so all such helpers were moved to file/anonymous-namespace scope instead.
- **Current measured size:** Host-proxy probe (control vs. a realistic `signal<button_event,4>` with 2 listeners attached — 1 free function, 1 member function — plus one `publish()` call, both built under the real aggressive flag set on this machine's Clang/llvm-mingw): **+264 bytes total (+192 .text, +72 .data)** over the empty-`main()` baseline. This is the first size number this project has that reflects real dispatch logic rather than being fully constant-folded away (delegate alone, last firing, collapsed to zero overhead when statically knowable — signal's runtime-varying multi-listener loop can't be proven-constant the same way, so this is a genuine, if still host-proxy, measurement). Caveats, stated plainly rather than overclaimed: (1) this is x86-64 Windows PE/Clang, not any of the actual 32-bit embedded targets — pointers are half the size there and the ISAs (Thumb-2, RISC-V C-extension, Xtensa) are generally denser, so the real embedded number will very likely be smaller, not larger, but that's not yet measured, only reasoned; (2) this covers exactly one event type / one signal instance — per the deferred-core-extraction note above, a second, third, etc. distinct event type would each add a full independent copy of this cost under the current fully-templated design, not a small marginal shim cost, until/unless the core-extraction refactor happens; (3) `size_audit/`'s proper CMake-driven host/embedded pipeline still isn't wired up (same gap noted last firing) — this was a direct ad hoc `clang++`/`llvm-size` probe in the scratchpad, not the checked-in tooling. 264 bytes for one event type with 2 listeners is comfortably inside the 1-2KB budget with real headroom.
