- **Last completed:** Picked up last firing's candidate (a): now that firings 16-18 produced real multi-event-type size data and a mechanism-level understanding (`attach()` never shares code across event types; `publish()` mostly does, automatically, via the compiler's own IPA-ICF pass), formally revisit `Research/ARCHITECTURE.md`'s Phase-0 amendment, which explicitly deferred the `detail::signal_core` shared-core extraction pending exactly this kind of data. This was treated as a genuine architectural fork, not a mechanical next-step — dispatched 2 subagents in parallel (per the loop's investigate-before-implementing rule) rather than deciding solo:
  1. **Pressure-test**: given the real data, has the amendment's own trigger condition ("revisit once there's real multi-event-type size data showing the naive per-type cost is actually a problem") actually been met? Verdict: no — 375 bytes for 5 event types leaves ~650-1,670 bytes of headroom against the 1-2KB budget, comfortable for a project with no real external consumers yet. Recommended a middle path: don't do the full rewrite (would re-risk the already-proven-safe `publish()`/dispatch reentrancy contract for a benefit — `publish()` sharing — that mostly already exists via IPA-ICF), but `attach()`'s 100%-duplicated cost is real and uncompensated by anything the compiler does automatically.
  2. **Narrower-design investigation**: is there a lower-risk alternative to the original full event-erasure design, scoped to just `attach()`/`detach()`'s Event-agnostic slot-scan (which never touches `Event`'s type at all — only `delegate<Sig>`'s fixed 2-pointer layout)? Verdict: yes, and technically sound — a `slot_view{void* a; void* b;}`-based shared finder (`find_empty`/`find_matching`/`clear`), built via `memcpy` (not `reinterpret_cast`, to stay inside defined behavior) with a *runtime* slot count, careful never to interpret a stub-pointer word as a meaningful address (relevant for AVR's Harvard architecture, a real in-scope target). Found something neither prior firing had made explicit: the duplication axis is per-(Event, MaxListeners, Fn), not just per-Event — two different listeners on the *same* signal already duplicate the whole scan today. Recommended trying a much cheaper experiment first: mark `attach()`/`detach()` `[[gnu::noinline]]` and see if the same automatic IPA-ICF folding that already helps `publish()` picks these up too, before hand-rolling anything.
  Both subagents converged on the same practical shape (defer the full rewrite; the narrow `attach()`-only path is the real candidate if any), so the loop **ran the cheap experiment for real rather than debating it further**: added `[[gnu::noinline]]` to `attach<Fn>()`, `attach<MemFn,T>(T*)`, `detach<Fn>()`, `detach<MemFn,T>(T*)`, and `detach_matching()` in `include/lweh/signal.hpp`, rebuilt, and measured on real Xtensa hardware. **Result: 437 bytes — a 62-byte regression against the 375-byte baseline, not an improvement.** The `nm` output explained why immediately: each `attach<Fn>()` instantiation embeds a genuinely different literal constant (the specific `Fn`'s `free_function_stub` address), so the four instantiations are not byte-identical and IPA-ICF correctly can't fold them — forcing them out-of-line only added call/return overhead with zero offsetting sharing (`$constprop$4`-style GCC clone-symbol suffixes visible in the `nm` dump, one per distinct `Fn`). **Reverted** (`git checkout -- include/lweh/signal.hpp`), confirmed the revert restores exactly 375 bytes. This is a real, decisive, informative negative result, not a wasted detour: it rules out the cheap fix and confirms the narrower hand-rolled shared-finder design (which explicitly separates the Fn-independent scan from the Fn-dependent write) is the *only* thing that could actually capture this savings, if it's ever worth building. Recorded the full decision, including the now-fully-specified narrow-finder design, in `Research/ARCHITECTURE.md`'s amendment section so a future firing doesn't have to re-derive any of this. **No code change lands from this firing** — `signal.hpp` is back to its pre-firing-19 state, byte-for-byte.
  **Also corrected a real, minor inaccuracy** in firing 18's own PROGRESS.md text, caught by the pressure-test subagent: it described "730 bytes total for 5 event types" as if that were the size figure to weigh against the 1-2KB budget. 730 is the *raw absolute* with-lweh binary size (including the ~355-byte platform floor); the correct, budget-relevant figure — used consistently everywhere else in this ledger — is 375 bytes (with-lweh minus control). Both numbers are individually correct (730 − 355 = 375, arithmetic checks out), but stating 730 as "the total" in that one sentence was misleading phrasing, not an arithmetic error. Fixed here.
