But that pass reached `cmake/LwEhConfig.cmake.in` and found something genuinely different in kind: real, substantial CMake packaging infrastructure (`install(TARGETS lw_eh EXPORT LwEhTargets ...)`, `install(EXPORT LwEhTargets NAMESPACE LwEh:: ...)`, `configure_package_config_file()`, `write_basic_package_version_file()`) that's existed since Phase 0, is explicitly documented in research.md §B1 as a legitimate consumption path, and had **never once been exercised end-to-end** — every other consumption path this repo actually tests (`tests/`, `examples/`, `size_audit/`) uses `add_subdirectory`, which never touches `install()`/`find_package()` at all. A regression anywhere in that packaging machinery (a typo in the EXPORT name, a wrong `INSTALL_DESTINATION`, a missing `FILE_SET`) would have been invisible to every check in this repo.
  Verified it manually first (scratch prefix, `cmake --install`, a throwaway consumer project doing `find_package(LwEh REQUIRED)` + `target_link_libraries(... LwEh::lw_eh)`, actually running the resulting binary, not just linking it) — confirmed working end-to-end, correct dispatch through both storage policies. Then committed the consumer as a permanent, real fixture (`packaging_check/{CMakeLists.txt,main.cpp}`) rather than leaving the verification as a one-off: deliberately **not** `add_subdirectory`'d from the root (structurally circular — can't `find_package()` a package still mid-configure), so it's a standalone project by design, documented as such in its own `CMakeLists.txt`. Re-verified the *committed* files work identically to the scratch version before trusting them. Added a new `packaging-check` CI job exercising the same flow on `ubuntu-latest` — this one clears the bar for permanent CI investment (unlike firing 24's compile-fail-test question, which didn't): real external-facing documented contract, genuinely complex multi-command machinery with no other coverage at all, robust check categories (configure/build/run succeeding, not fragile compiler-message substring matching), closer to the `host-asan` precedent than the `signal_core`/compile-fail-test one.
  Two smaller things caught and fixed along the way, both while doing this work rather than as separate detours: (1) the hermeticity-lint job's grep scope correctly excludes `packaging_check/` (that directory's `find_package(LwEh)` is a downstream consumer resolving Lw-Eh *as* a dependency — the exact legitimate use case being verified — not Lw-Eh resolving something for itself), but this was only true by accident of the existing directory allowlist, not documented anywhere; added an explicit comment so a future reader doesn't mistake it for an oversight. (2) `.gitignore`'s `/build/` pattern is root-anchored and would **not** have caught `packaging_check/build/` — exactly the directory `packaging_check/CMakeLists.txt`'s own usage instructions tell people to build into — verified the gap empirically (`git check-ignore` before/after) and generalized the pattern to be depth-independent (`build/`, no leading slash), confirmed both the existing root build dirs and the new nested case are now correctly ignored. Also updated README's "Integration" list (which had never mentioned `find_package` as an option at all, alongside vendoring/`add_subdirectory`/ESP-IDF/PlatformIO) and "Building and testing" section to document the new fixture and CI job.
