- **Last completed (firings 38-46, real-hardware-validation thread, now closed):** A real ESP32 board became available (COM5, CP210x bridge) — the first time this project could validate on actual silicon instead of only host tests + static cross-compiled size measurement. Getting `examples/esp32_minimal` to genuinely boot and run correctly required finding and fixing **four distinct, real, hardware-only bugs** that no amount of host testing or static analysis alone caught, each confirmed via real disassembly/exception decoding, not guessed:
  1. `.bss`/`.data` byte-accessed globals were placed in IRAM0 (the word-only-accessible instruction bus) — a real `LoadStoreError` on a byte-store. Fixed by splitting IRAM0 (code) from DRAM0 (data) in `esp32.ld` (commit `e438462`).
  2. `.rodata` (specifically string literals, read byte-by-byte) had the identical IRAM0-byte-access problem — vtables in the same section were fine since they're word-accessed. Fixed by moving `.rodata` to DRAM0 too (commit `ac83160`).
  3. **A genuine `esptool.py elf2image` limitation**: it builds image segments from each section's VMA alone and does not implement a linker script's `AT>` load-address (LMA≠VMA) mechanism at all — the textbook "copy `.data` from flash to RAM at boot" pattern silently never got anything written to the intended load address, so the runtime copy loop was overwriting correctly-loaded DRAM0 content with uninitialized IRAM0 garbage. Fixed by dropping the `AT>` split entirely — the bootloader's own segment loader already places each section's bytes directly at its VMA, no copy needed (commit `1d40989`). Two independent investigation threads reached this same diagnosis in parallel.
  4. A crash immediately after `app_main()` returned, landing in the bootloader's own memory — `app_main()` is entered via a non-windowed `call0` into windowed-ABI (`entry`/`retw`) code, and the one problematic transition was its own outermost `retw.n` crossing back over that boundary with no window-exception vectors set up. Fixed the simple, idiomatic way (real vector-table code would be substantial, unnecessary work here): `app_main()` never returns, looping forever after its work is done — exactly how a bare-metal entry point with no OS to return to should behave anyway. **Independently cross-validated at the ISA level by a second investigation** (real citations: Espressif's `WindowedOption.tex`, ESP32's actual `core-isa.h`/`xtensa_vectors.S`): `entry` unconditionally rotates the register window from whatever `PS.CALLINC` holds with no check a matching windowed `callX` preceded it, and `call0` never sets `PS.CALLINC` — confirming this is architecturally the *only* broken transition (every call/return nested inside `run_scenario()` is self-consistent by construction) and that real window-exception vectors, while cheap to add if ever needed, aren't needed here. This unmasked a second, separate issue: the bootloader's own ~9s RTC watchdog, never disabled by this from-scratch app, firing regardless of correct execution. Fixed via `examples/esp32_minimal/rtc_wdt.hpp` (real, cited register addresses) (commit `b6db5a7`).
