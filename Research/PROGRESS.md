# Lw-Eh Progress Ledger

This is the durable, git-tracked ledger for the autonomous build loop. Updated at the end of every firing. Git history is ground truth if this ever drifts — see `git log --oneline`.

- **Phase:** core-implementation (both storage policies now implemented and tested; real embedded validation remains the main gap)
- **Last completed:** Implemented `lweh::intrusive_node<Event>`/`lweh::intrusive_signal<Event>` (`include/lweh/intrusive_signal.hpp`, `include/lweh/detail/intrusive_core.hpp`) — the alternate, unbounded, zero-allocation storage policy from research.md §A7. Hand-derived the singly-linked, head-insertion design (attach = O(1) prepend to `head_`, detach = O(n) scan-and-relink, dispatch = forward walk) and traced through the reentrancy scenarios myself before writing code, then had 1 subagent adversarially stress-test the hypothesis with compiled, ASan/UBSan-checked programs (matching the pattern that worked well for `signal<Event,N>`'s reentrancy design two firings ago) rather than a from-scratch prior-art comparison, since last firing's EnTT/Linux-kernel/ETL research already directly informs this case (no new comparison needed). Result: self-detach, detach-any-other (visited or not, adjacent or not), and attach-during-dispatch (confirmed **unconditionally deferred to the next pass regardless of timing** — simpler and more predictable than `signal<Event,N>`'s slot-index-dependent contract) are all safe. But the stress test **found a real, confirmed, reproducible bug**: if a listener self-detaches AND ALSO detaches a different listener that was its own not-yet-visited immediate successor at that moment, in that order within one callback, the second listener spuriously fires once more that same pass despite its `detach()` call having already legitimately returned `true`. Root cause: `unlink()` deliberately never touches the *removed* node's own `next` field (that's what makes plain self-detach safe at all) — when this exact compound sequence happens, that field becomes a stale snapshot the dispatch loop's cursor still trusts. Confirmed the classic "capture next before invoking" fix (Linux kernel style) does *not* help here — the staleness isn't from mutation-during-read, it's from a second, independent removal the frozen field was never told about. A real fix would need a generation counter or re-scan-on-mutation mechanism; given the bug is narrow, order-dependent, always avoidable by reordering the two detach calls, self-healing by the very next pass, and not a memory-safety issue (confirmed clean under ASan/UBSan), documented it clearly in both `detail/intrusive_core.hpp`'s doc comment and a dedicated pinning test (`tests/intrusive_signal_test.cpp` scenario 6) rather than adding complexity to eliminate it — consistent with how `delegate::operator==`'s narrow ICF-folding caveat was handled two firings ago, not a new pattern. Wrote 6 real test scenarios total (basic LIFO dispatch order, detach/no-op-detach, self-detach, detach-not-yet-visited-peer, attach-during-dispatch timing-independence, the known-caveat pin) — all pass on the first try, confirming the hand-traces were correct. `intrusive_signal`/`intrusive_node` use plain runtime references (no `template<auto Fn>` NTTP anywhere), so — unlike `signal<Event,N>`'s tests — local classes inside `main()` are genuinely safe here; used them freely without the linkage concern from three firings ago.
- **Current measured size:** Host-proxy probe (no `-flto`, matching the more representative embedded-default case per research.md's toolchain findings): 1 event type via `intrusive_signal` costs **+216 bytes** over the empty-`main()` baseline; a 2nd event type costs **+168 bytes marginal** (384 total − 216) — roughly 78% of the first type's cost, i.e. *not* dramatically better than `signal<Event,N>`'s own ~54% ratio measured two firings ago. This tempers the initial expectation that virtual-dispatch erasure would deliver a big, obvious win over the deferred-core-extraction array design. Investigated why via `llvm-nm` symbol inspection (not just size deltas) on an unstripped build: confirmed `lweh::detail::intrusive_core::dispatch(void const*) const` appears as **exactly one symbol** regardless of event-type count — the shared dispatch loop genuinely is compiled once, exactly as designed, with `link`/`unlink` fully inlined away entirely (even better than staying as separate shared functions). The per-type cost that *isn't* shared and can't be, under any design: one `invoke_erased` adapter per Event type (does real type-specific casting, unavoidably different code per type) and one vtable per concrete listener class. So the design delivers exactly what it promised for the piece research.md §A4 was actually worried about (the dispatch loop itself doesn't duplicate), but that piece turns out to be a smaller slice of the real per-type cost than vtables/adapters/callback-bodies, which are inherent to supporting a new event type under *any* storage policy — not a design failure, a more complete and honest picture than the size-delta numbers alone would suggest. Same standing caveats as prior firings: x86-64 Windows PE host-proxy data, not the real 32-bit embedded ISAs; `size_audit/`'s real CMake wiring is still unfixed and non-urgent.
