- **Last completed:** Fresh orientation noticed firing 23's three new `bind()` compile-time checks (one pre-existing, two newly added — see `include/lweh/delegate.hpp`) had never actually been exercised by anything: the whole CTest suite only ever compiles *correct* bindings, so a future edit that silently weakened or deleted a `static_assert` would not break any currently-passing test. Treated "should this be automated as a standing CTest mechanism" as a genuine engineering decision (a brand-new testing category this codebase has never used) and dispatched 2 parallel subagents: one on the pure CMake/CTest mechanics of testing "this snippet must fail to compile with this exact message," one on whether the investment is justified at all.
  The mechanics investigation was unusually valuable precisely because it *disproved* the naive approach empirically rather than just describing it: `try_compile()`, the obvious first instinct, works from a real project context but — verified via a real reproduction, not reasoned abstractly — **does not track headers a probed source `#include`s as a reconfigure dependency**. Editing only the *header* (i.e. `delegate.hpp` itself — the realistic regression this whole feature exists to catch) leaves a stale, wrongly-"Passed" cached result in place with no reconfigure firing. Implemented naively, this would have shipped something that *looked* like protection but silently wasn't, for the exact scenario it was built to catch. The subagent's recommended fix (direct compiler invocation via a `cmake -P` driver script + `execute_process()`, a fresh subprocess every `ctest` run, nothing cached to go stale) is real, complete, and fully verified end-to-end on this project's exact Windows/Ninja/Clang setup, including the negative cases (correctly detects both an unexpected compile success and a right-failure-wrong-message case).
  The evaluation investigation recommended against building it anyway, and — weighed independently, not just deferred to — that argument held up even against the clean mechanics: each of the 3 checks protects exactly one self-contained line with zero historical incidents; the mechanism doesn't generalize for free (a future static_assert elsewhere still needs its own hand-written scenario, the same as any other test category); and the sharpest point, static_assert *message text* is compiler prose, not a stable contract — a future clang version rewording the exact phrasing would break a substring-matched test for reasons unrelated to any actual regression, exactly the kind of non-actionable churn this project has explicitly rejected elsewhere. Benchmarked explicitly against two precedents: closer in shape to firing 19's deferred `signal_core` rewrite (narrow, no measured urgency) than firing 21's `host-asan` preset (broad, diffuse, proven-real historical gap across the whole test surface, near-zero marginal cost).
  **Verified all three checks properly instead of building permanent infrastructure**, using the mechanics investigation's own proven-correct approach (direct `clang++ -std=c++17` invocation) applied for real against the actual, current `include/lweh/delegate.hpp` (not a scratchpad reproduction) on the project's pinned compiler:
  - Control (valid free-function and member-function bindings): compiled clean, as expected.
  - `bind<&widget::tick>()` where `tick()` is a zero-arg method (deliberately shaped, per the mechanics investigation's own finding, to also satisfy the *new* `is_invocable_r_v` check via the documented INVOKE-unbound-member loophole, so this genuinely exercises the intended backstop rather than being caught by the newer check first): failed with exactly `"bind<Fn>() is for free functions only; use bind<MemFn>(instance) for member functions."`, instantiation trace confirming it passed through `bind<Fn>()` untouched and was caught inside `free_function_stub`, precisely as designed.
  - A free function with the wrong arity bound via `bind<Fn>()`: failed immediately at `bind<Fn>()`'s own line with exactly `"bind<Fn>(): Fn is not callable as Ret(Args...)."` — the new check firing early, before the stub is even instantiated, exactly the diagnostic-quality improvement it was added for.
  - A member function with the wrong arity bound via `bind<MemFn>(T*)`: failed with exactly `"bind<MemFn>(instance): MemFn is not callable as Ret(Args...) on *instance."` — the previously totally-unguarded overload's new check, confirmed working for the first time.
  All four snippets and their compiled `.o` files were scratchpad-only and deleted after verification; `git status` confirms the repo itself is untouched. **Explicit revisit trigger, matching this project's established pattern (see `Research/ARCHITECTURE.md`'s `signal_core` amendment) rather than leaving this open-ended**: automate this as standing CTest infrastructure if `delegate.hpp` ever starts accepting changes without a human reading the diff (CI auto-merge, external contributors), or if a real near-miss regression actually happens — not before, and not just because the mechanism turned out to be cheap to build correctly.
