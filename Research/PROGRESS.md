This is the durable, git-tracked ledger for the autonomous build loop. Updated at the end of every firing. Git history is ground truth if this ever drifts — see `git log --oneline`.

- **Phase:** embedded-validation (unchanged — real Xtensa hardware measurement, no new library feature)
- **Last completed:** Picked up candidate (1) from last firing's handoff: measure the marginal size cost of an additional `signal<Event,N>`-backed event type, to start validating research.md §A10's core claim ("shared engine + small marginal per-event-type cost, not a naive flat cap") with real data instead of the one aggregate number that's been on record since firing 8. Added a fourth event type, `battery_level_event` (free-function listener, mirroring `button_event`'s shape), to the shared `scenario.hpp` used by both the embedded example and the size probe — deliberately a **third `signal<>`-backed type**, not a second `intrusive_signal<>` one, because `signal<>` has no shared dispatch core (each instantiation compiles its own full attach/detach/publish logic — see `detail/signal_core.hpp`'s comment, corrected last firing) while `intrusive_signal<>` already does, making `signal<>`'s marginal cost the real open question.
  **A real, important correction surfaced while measuring**: the previously-recorded 520-byte number (on record since firing 8, carried forward unchanged through firings 10-15) turned out to be **stale**, not current. It predates firing 11's `-Wl,--relax` addition to `lw_eh_size_flags` — that firing verified `--relax` didn't break the *host* `lld` build, but the real embedded Xtensa number was never re-measured on real hardware afterward. Every firing since correctly carried the number forward unchanged (none of them touched compiled/embedded code, so re-measuring would have violated the proportionality rule) — no single firing was wrong, but the number quietly went stale for four firings as a result. This firing is the first real-hardware measurement since that flag landed. To isolate the *true* marginal cost of the new event type from this flag-related drift, rather than conflating the two: `git stash`'d this firing's changes, rebuilt+remeasured the *original* 3-event-type scenario under *today's* flags (86 bytes incremental — not 520), `git stash pop`'d, rebuilt+remeasured again with the new 4th type (229 bytes incremental, reproduced twice, identical both times). Both numbers come from the same build directory, same cache, same toolchain, measured back-to-back — a clean, controlled comparison, not a before/after across drifted conditions.
- **Current measured size:** **229 bytes incremental** on real ESP32 Xtensa hardware (`xtensa-esp32-elf-size`, 2026-07-16, current `lw_eh_size_flags` including `-Wl,--relax`), for **four** event types (button_event, sensor_reading_event, battery_level_event on `signal<>`; connection_event on `intrusive_signal<>`) — supersedes the stale 520-byte number. **Marginal cost of the new (4th) event type: 143 bytes** (229 − 86, both measured under identical current conditions). Broken down via `nm --size-sort` diff between the two runs: 32 bytes for the `signal<battery_level_event,4>` object itself (4 empty delegate slots × 8 bytes, matching README's stated per-slot cost), 27 bytes for the `free_function_stub<&on_battery_level>` trampoline (the actual "no shared core" tax `signal_core.hpp`'s deferred-extraction note warned about), 1 byte for the new flag variable, and the remaining ~83 bytes from `app_main`'s own growth (the new `attach<>()`/`publish()` call sites for the 4th type, inlined into the entry point). Honest assessment against §A10's "small" framing: 143 bytes for one more type is real, not negligible — at that rate, roughly 7-14 more `signal<>`-backed event types would consume the entire stated 1-2KB budget on their own, before accounting for anything else. Not alarming for a project this early with this few real consumers, but a concrete number to weigh next time someone asks whether the deferred `signal_core` extraction is worth doing yet.
